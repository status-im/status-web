In <a href="https://docs.status.network/overview/general-info/gasless-transactions"><strong>Rate Limiting Nullifiers</strong></a><strong> (RLN)</strong>, this property enforces rate limits without exposing user identity.</p><p>Exceed your quota and the system rebuilds your secret key. Stay within it and you remain private.</p><h2 id="what-is-shamirs-secret-sharing">What Is Shamir's Secret Sharing?</h2><p>Cryptographer Adi Shamir introduced <strong>Shamir's Secret Sharing (SSS)</strong> in 1979. It solves a clear problem.</p><p>How do you protect a secret so no single party holds all the power? And how can a group still recover it when needed?</p><p>The scheme rests on a simple math principle.</p><h3 id="how-the-math-works">How the Math Works</h3><p>SSS uses polynomial interpolation over a finite field.</p><p>To split a secret <code>S</code> among <code>n</code> parties with a threshold of <code>k</code>:</p><ol><li>Build a polynomial of degree <code>k-1</code>. Set the constant term equal to <code>S</code>.</li><li>Evaluate the polynomial at <code>n</code> different points.</li><li>Give one point (share) to each party.</li></ol><p>Any <code>k</code> parties can combine their shares using <a href="https://en.wikipedia.org/wiki/Lagrange_polynomial"><strong>Lagrange interpolation</strong></a> to rebuild <code>S</code>. Fewer than <code>k</code> shares reveal nothing. This is a <code>(k, n)</code> threshold scheme.</p><p>A quick example: a <code>(3, 5)</code> scheme splits a secret into 5 shares. Any 3 rebuild the secret. Any 2 learn nothing.</p><h3 id="why-this-is-strong">Why This Is Strong</h3><p>SSS offers <strong>information-theoretic security</strong>. Security does not depend on computational hardness.</p><p>An attacker with unlimited computing power still cannot recover the secret from fewer than <code>k</code> shares. The math makes it impossible, not just hard.</p><h2 id="what-are-rate-limiting-nullifiers">What Are Rate Limiting Nullifiers?</h2><p><strong>Rate Limiting </strong><a href="https://status.network/blog/what-is-a-nullifier-zero-knowledge-cryptography"><strong>Nullifiers </strong></a><strong>(RLN)</strong> is a zero-knowledge protocol. It enforces per-user rate limits in anonymous systems.</p><p>It solves a tough challenge: how do you stop spam without knowing who the user is?</p><p>Classic anti-spam tools rely on identity. You block accounts or blacklist IP addresses. Those methods break anonymity.</p><p><strong>RLN</strong> takes a different path. It uses cryptographic proofs and secret sharing to punish quota violations with math, not a central authority.</p><h3 id="the-core-mechanism">The Core Mechanism</h3><p>Each user registers a secret key in a shared Sparse Merkle Tree. This structure allows the protocol to verify that a user belongs to the network without revealing exactly which user they are.</p><p>When a user sends a transaction, they produce a <strong>zero-knowledge proof (ZKP)</strong> that:</p><ol><li>Proves their secret key is registered in the tree.</li><li>Encodes a share of their secret key into the proof output.</li><li>Does not reveal the secret key itself.</li></ol><p>Each valid action produces one share. Stay within the rate limit and your secret is never rebuilt. Each proof reveals a different share tied to your key and the current epoch.</p><h3 id="where-shamirs-secret-sharing-enters">Where Shamir's Secret Sharing Enters</h3><p>Here is where SSS becomes the enforcement tool.</p><p>The shares in RLN proofs are structured as points on a polynomial. The user's secret key is the constant term. This matches standard SSS exactly.</p><p>If a user submits more actions than the allowed rate, different shares from the same epoch pile up on-chain. Once enough shares exist (meeting the threshold <code>k</code>), anyone can run Lagrange interpolation. The result is the user's secret key.</p><p>The recovered key serves as cryptographic proof of a violation.
