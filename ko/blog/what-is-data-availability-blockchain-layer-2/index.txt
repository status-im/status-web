28:T25af,<p><strong>Data Availability (DA)</strong> is the cryptographic guarantee that all transaction data within a blockchain block is fully accessible for download and verification. Without DA, Layer 2 (L2) rollups cannot prove their state transitions are honest. If transaction data is withheld, verifiers cannot challenge fraudulent batches or confirm valid ones, breaking the trustless nature of the network.</p><h2 id="why-does-data-availability-matter">Why Does Data Availability Matter?</h2><p>Blockchains function through decentralized verification: anyone can "check the math". If a block producer publishes a valid-looking block header but withholds the underlying transaction data, it creates a <strong>Data Availability Problem</strong>. This is not about long-term storage, but immediate <strong>access</strong>.</p><p>For Layer 1s like Ethereum, redundancy solves this, thousands of nodes hold copies. For Layer 2 rollups, which execute transactions off-chain to boost throughput, DA is the critical link that inherits the security of the mainnet</p><h2 id="how-rollups-create-a-da-dependency">How Rollups Create a DA Dependency</h2><p>A rollup executes transactions off Ethereum mainnet. It then posts compressed results back to Ethereum. This design cuts costs and boosts throughput.</p><p>But rollups inherit Ethereum's security only if one condition holds: the underlying data must be available.</p><p>Here is the logic, step by step.</p><h3 id="step-1-batch-execution">Step 1: Batch Execution</h3><p>The rollup <strong>sequencer</strong> collects user transactions. It executes them in order. It produces a new state root reflecting all balance changes.</p><h3 id="step-2-data-posting">Step 2: Data Posting</h3><p>The sequencer posts transaction data (or a compressed form) to a DA layer. For most rollups today, that layer is Ethereum itself.</p><h3 id="step-3-proof-or-challenge">Step 3: Proof or Challenge</h3><p><strong>Optimistic rollups</strong> assume batches are correct. Verifiers have a window (often 7 days) to submit fraud proofs if something is wrong. They need the original data to construct those proofs.</p><p><strong>ZK rollups</strong> submit validity proofs (zero-knowledge proofs) alongside each batch. The proof confirms correct execution. But validators still need data access to verify the proof inputs match real transactions.</p><h3 id="what-happens-if-da-fails">What Happens if DA Fails?</h3><p>If transaction data is withheld, fraud proofs become impossible. A dishonest sequencer could steal funds. No one could prove the theft occurred.</p><p>For ZK rollups, withheld data means users cannot reconstruct their own balances. They cannot exit the rollup. Their funds become trapped.</p><p>DA failure breaks the entire security model.</p><h2 id="where-does-da-data-actually-live">Where Does DA Data Actually Live?</h2><p>Rollups have several options for posting their data. Each involves tradeoffs between cost, security, and decentralization.</p>

<h3 id="ethereum-calldata">Ethereum Calldata</h3><p>The original approach. Rollups embed data directly in Ethereum transaction calldata. Every Ethereum node stores it. Security is maximal. Cost is also maximal.</p><h3 id="ethereum-blobs-eip-4844">Ethereum Blobs (EIP-4844)</h3><p><strong>EIP-4844</strong> introduced "blob" transactions in 2024. Blobs provide temporary, cheaper DA space on Ethereum. Data persists for roughly 18 days. This is long enough for fraud proof windows.</p><p>Blobs reduced DA costs by over 90% for many rollups.</p><h3 id="dedicated-da-layers">Dedicated DA Layers</h3><p>Projects like Celestia and EigenDA offer specialized DA services. They run separate validator networks tuned for data throughput. Rollups using these layers gain cost savings. They accept a different security model.</p><h3 id="off-chain-da-validium">Off-Chain DA (Validium)</h3><p>Some systems store data with a small trusted committee. Costs are minimal. But security depends on that committee not colluding. This is the weakest DA guarantee.</p><h2 id="data-availability-sampling-the-scaling-frontier">Data Availability Sampling: The Scaling Frontier</h2><p>Full nodes download all data. This does not scale forever. <strong>Data availability sampling</strong> (DAS) offers a solution.</p><p>With DAS, light nodes download only small random chunks of a blob. If enough random samples succeed, the node gains high confidence that the full data exists.</p><p>DAS relies on a math technique called <strong>erasure coding</strong>. The original data is expanded with redundancy. Even if 50% of the coded data is missing, the full original can be reconstructed.</p><p>Ethereum's long-term roadmap (sometimes called "Danksharding") plans to use DAS. This would dramatically increase blob capacity without requiring every node to download everything.</p><h2 id="how-status-network-handles-da">How Status Network Handles DA</h2><p><a href="https://status.network/"><strong>Status Network</strong></a> is an Ethereum Layer 2 built on the <a href="https://linea.build/"><strong>Linea</strong></a> zkEVM rollup stack.
