21:T2a3b,<p>nullifier is a unique tag derived from a secret key. It marks that an action already happened. ZK protocols reject any duplicate nullifier, stopping double-spends without revealing who acted. Each valid action produces exactly one nullifier. This guide explains the key.</p><h2 id="why-double-spending-is-hard-to-solve-in-private-systems">Why Double-Spending Is Hard to Solve in Private Systems</h2><p>Public blockchains solve double-spending with open data. Every node sees every balance. If Alice spends 1 ETH, the full network records it.</p><p>Privacy breaks that model. When balances are hidden, the network cannot tell if Alice spent the same coin twice.</p><p>One fix: reveal the sender. But that kills privacy.</p><p>Nullifiers offer a better path. They prove an action happened once. They never reveal who did it.</p><h2 id="how-a-nullifier-works-step-by-step">How a Nullifier Works Step by Step</h2><p>A nullifier is not random. It is computed from inputs only the user controls. The process follows a strict order.</p><ol><li>The user holds a secret (a private key or commitment).</li><li>The user builds a ZK proof that the secret is valid.</li><li>A nullifier hash is derived from that secret.</li><li>The proof and nullifier are sent together.</li><li>The verifier checks the proof. Then it checks the nullifier against a stored set.</li><li>A new nullifier means success. A known nullifier means rejection.</li></ol><p>The key rule: the same secret always yields the same nullifier. You cannot create a second nullifier from one secret. One secret, one nullifier, one valid action.</p><h2 id="what-makes-nullifiers-deterministic">What Makes Nullifiers Deterministic?</h2><p>Determinism is the core design choice. The nullifier is a hash of fixed inputs. Same inputs always produce the same output. A simple formula looks like this:</p><p><code>$$nullifier = Poseidon(secret, externalNullifier)$$</code></p><p>The <code>$externalNullifier$</code> ties the action to a specific scope, such as an epoch. Alice can act once per scope. She cannot reuse her secret for a second action in that scope. The math blocks it.</p><h2 id="nullifiers-in-private-transactions">Nullifiers in Private Transactions</h2><p><strong>Zcash</strong> pioneered nullifiers for private payments. Each shielded note has a unique commitment. When spent, the owner reveals the nullifier, not the note.</p><p>The blockchain stores every spent nullifier. Nodes check this set before accepting a transaction. A match means the note was already used. The system rejects it.</p><p>The owner proves three things in zero knowledge. The note exists in the commitment tree. The nullifier matches the note. The owner holds the spending key. No amounts or addresses are exposed.</p><h2 id="nullifiers-in-anonymous-voting">Nullifiers in Anonymous Voting</h2><p>Voting protocols face the same puzzle. Each voter should cast one ballot. But voter identity must stay hidden.</p><p>A nullifier scoped to the election ID solves this. Each voter derives one nullifier per election. The tally contract rejects any duplicate.</p><p>The voter proves eligibility with a ZK proof. The ballot counts. The voter stays private.</p><h2 id="nullifiers-beyond-double-spend-rate-limiting">Nullifiers Beyond Double-Spend: Rate Limiting</h2><p>Nullifiers do more than block reuse. They can also enforce rate limits.</p><p><strong>Rate Limiting Nullifiers (RLN)</strong> extend the basic model. Instead of one action per scope, <strong>RLN</strong> allows a set number per epoch. A user might send five messages per time window.</p><p>The design is clever. <strong>RLN</strong> pairs <strong>Shamir Secret Sharing</strong> with ZK proofs. Each action in an epoch reveals one share of the user's secret.</p><p>Stay within the limit, and your secret is safe. Go over, and enough shares are exposed to rebuild your secret key.</p><p>This builds a penalty into the protocol. Spammers lose their keys. Honest users stay anonymous.</p><h3 id="how-rln-uses-sparse-merkle-trees">How RLN Uses Sparse Merkle Trees</h3><p><strong>RLN</strong> tracks members in a <a href="https://status.network/blog/sparse-merkle-tree-smt-rate-limiting"><strong>Sparse Merkle Tree</strong></a> of height 20. This supports up to 1 million accounts. Each member holds a leaf in the tree.</p><p>To act, a member proves their leaf exists. They never reveal which leaf is theirs. The nullifier confirms the action is unique within the epoch.</p><p>A <strong>Deny List</strong> catches rule breakers. Users who exceed their quota lose free access and must pay premium fees to continue transacting.</p><h2 id="how-status-network-uses-rln-for-gasless-transactions">How Status Network Uses RLN for Gasless Transactions</h2><p>Status Network is a gasless Ethereum Layer 2 built on Linea's zkEVM stack. It swaps gas fee markets for a reputation model powered by RLN. Every user with positive <strong>Karma</strong> (a soulbound reputation token) gets a free transaction quota. RLN nullifiers enforce that quota at the protocol level.</p><ol><li>A user earns Karma by staking SNT, bridging assets, adding liquidity, or building apps.</li><li>Their Karma balance sets the free transaction tier.</li><li>Each transaction creates an RLN nullifier scoped to the current epoch.</li><li>The system tracks nullifiers. Quota-compliant transactions cost nothing.</li><li>Transactions exceeding the quota trigger the Deny List, and the user must pay a premium sequencing tip.</li><li>If a user violates the global rate limit, their secret is recovered and they face reputation slashing.</li></ol><h3 id="why-this-matters-for-gasless-l2-design">Why This Matters for Gasless L2 Design</h3><p>Most L2s charge gas to stop spam. <a href="https://status.network/blog/status-network-economic-model-yield-funding-pool-and-zero-gas">Status Network</a> proves another way exists. ZK nullifiers enforce rate limits instead. Native yield from bridged assets funds the network. No gas fees for compliant users. Nullifiers make it work.</p><h2 id="nullifiers-and-on-chain-bot-economics">Nullifiers and On-Chain Bot Economics</h2><p>Nullifiers also shape how bots run on gasless networks. On a standard L2, bots pay gas like everyone else. <a href="https://status.network/blog/how-do-layer-2s-make-money">Gas cost</a> is the natural brake on bot spam.</p><p>On a gasless chain with <strong>RLN</strong>, bots face a different limit: their <strong>Karma</strong> transaction quota. A bot that burns through its free tier lands on the <strong>Deny List</strong>. It must then pay premium gas.</p><p>This filters out low-value spam bots. High-value bots (those catching DEX arbitrage on <strong>Orvex</strong> or running liquidations on <strong>FIRM</strong> CDPs) can justify the premium.</p><p>Bots that spread actions across epochs keep free access. Bots that burst past their limit pay the cost. Nullifier-based rate limits replace gas auctions as the primary bot-cost tool on reputation-based L2s.</p><h2 id="nullifiers-vs-nonces-what-is-the-difference">Nullifiers vs. Nonces: What Is the Difference?</h2><p>Both stop replay attacks. But they fit different trust models.</p>
