21:T27a1,<p>A Sparse <a href="https://status.network/blog/what-is-a-blockchain-bridge-cross-chain">Merkle Tree</a> (SMT) is a fixed-size cryptographic tree where most leaves are empty by default. It proves membership or non-membership of any value in constant time without scanning every entry.</p><p>In blockchain rate limiting, SMTs allow protocols to track user quotas and detect violations using <a href="https://status.app/blog/zero-knowledge-proof-zk-privacy-status">zero-knowledge proofs</a>, without exposing identity or requiring gas fees per check.</p><h2 id="what-is-a-merkle-tree">What Is a Merkle Tree?</h2><p>A Merkle Tree is a binary tree where each leaf holds a hash of data. Each parent node holds the combined hash of its two children. The root hash summarizes the entire dataset in a single value.</p><p>Change any leaf, and the root changes. This property makes Merkle Trees ideal for compact, tamper-evident proofs.</p><p>Standard Merkle Trees are built from actual data. You only store the leaves you have. The structure is dense and variable in size.</p><h2 id="what-makes-a-sparse-merkle-tree-different">What Makes a Sparse Merkle Tree Different?</h2><p>A Sparse Merkle Tree has a fixed, predetermined size. Every possible leaf position exists from the start, even if it holds no data.</p><p>Empty leaves store a known default value, typically a hash of zero. Non-empty leaves store actual data.</p><p>The key difference: you can prove that a leaf is empty just as easily as proving it is full. This is called a non-membership proof.</p><p>For a tree of height 20, there are 2^20 possible leaf positions, which equals exactly 1,048,576 slots. Most are empty. The tree is "sparse" because only a small fraction of leaves hold real values.</p><h2 id="why-height-20-matters-for-rln">Why Height 20 Matters for RLN</h2><p>Rate Limiting Nullifiers (RLN) on Status Network use an SMT of height 20. That number is not arbitrary.</p><p>Height 20 gives exactly 1 million possible account slots. This is the designed capacity for the system.</p><p>Each registered user occupies one leaf. The leaf stores a commitment derived from the user's secret key. The SMT root reflects the full state of all registered users at any point in time.</p><p>Provers generate <a href="https://status.network/blog/the-dawn-of-nano-transaction-economics-beyond-the-friction-of-gas">ZK proofs</a> against this root. Verifiers check those proofs without learning anything about the underlying secrets.</p><h2 id="how-does-rln-use-the-smt">How Does RLN Use the SMT?</h2><p>RLN is the spam-prevention mechanism powering gasless transactions on Status Network. Instead of charging gas per transaction, it enforces per-epoch message quotas using cryptographic rate limits.</p><p>Here is how the SMT fits into that system:</p><ol><li>Registration</li></ol><p>A user registers by inserting a cryptographic commitment into the SMT. The commitment is derived from a secret they hold locally. No identity is revealed on-chain.</p><ol start="2"><li>Proof Generation</li></ol><p>When sending a message, the user generates a ZK proof. The proof demonstrates two things: their commitment exists in the SMT (membership proof), and they have not exceeded their quota for the current epoch.</p><ol start="3"><li>Nullifier Output</li></ol><p>Each valid message produces a nullifier. Two messages in the same epoch from the same user will produce the same nullifier. This is the detection mechanism.</p><ol start="4"><li>Quota Violation</li></ol><p>If a nullifier appears twice, the protocol can reconstruct the user's secret key using Shamir's Secret Sharing. The user is then added to the Deny List.</p><p>The SMT is the foundation for step 2. Without efficient membership proofs, every quota check would require full tree traversal or expensive on-chain lookups.</p><h2 id="shamirs-secret-sharing-the-penalty-mechanism">Shamir's Secret Sharing: The Penalty Mechanism</h2><p><a href="https://en.wikipedia.org/wiki/Shamir%27s_secret_sharing">Shamir's Secret Sharing</a> splits a secret into shares. You need a minimum number of shares to reconstruct it.</p><p>In RLN, each message a user sends reveals one share of their private key. Sending within quota reveals only partial shares, which is not enough to reconstruct anything. Exceeding quota reveals enough shares to fully reconstruct the key.</p><p>Once the key is exposed, the system can slash or ban the user. This disincentivizes spam without requiring a central moderator or gas-based penalty.</p><p>The SMT enables this by making key commitment verification efficient inside a ZK circuit.</p><h2 id="zero-knowledge-proofs-inside-the-smt">Zero-Knowledge Proofs Inside the SMT</h2><p>Generating a ZK proof against an SMT works because the tree structure is deterministic.</p><p>For any given leaf position, the path from leaf to root is always the same length (equal to tree height). In a height-20 SMT, every proof path is exactly 20 hashes long.</p><p>This consistency is critical for ZK circuits. A circuit must have a fixed structure at compile time. Variable-length proof paths would require dynamic circuits, which are incompatible with current ZK proving systems like Groth16 or PLONK.</p><p>Fixed-height SMTs solve this. Every membership proof is the same size. Every circuit is the same shape.</p><h2 id="smt-vs-standard-merkle-tree-direct-comparison">SMT vs. Standard Merkle Tree: Direct Comparison</h2><h2 id="why-not-use-gas-fees-for-spam-prevention">Why Not Use Gas Fees for Spam Prevention?</h2><p>The traditional answer to spam on a blockchain is gas. If every transaction costs money, spamming becomes expensive.</p><p>Status Network rejects this model entirely. The network is designed to be gasless for users. Charging gas for spam prevention would contradict the core design.</p><p>RLN with SMTs provides a cryptographic alternative. Spam is disincentivized by the risk of key exposure, not by monetary cost. Users who stay within quota pay nothing and reveal nothing.</p><p>This aligns with the broader economic model: <a href="https://cryptoresearch.report/crypto-research/how-status-network-eliminates-ethereum-gas-fees/">native yield from bridged assets</a> (ETH converting to stETH, DAI converting to sDAI) funds network operations, while ZK-based rate limiting handles security.</p><h2 id="the-deny-list-what-happens-to-quota-violators">The Deny List: What Happens to Quota Violators</h2><p>Users who exceed their quota and have their key reconstructed are added to the Deny List. Their SMT commitment is marked invalid.</p><p>Future messages from that commitment will fail the membership proof. The ZK verifier rejects invalid commitments at the proof level, before any message is processed.</p><p>Importantly, a banned user is not permanently excluded. They can earn Karma back over time, which may restore their gasless transaction quota. Karma is a soulbound reputation token on Status Network, earned through staking SNT, bridging assets, providing liquidity, and other contributions.</p><p>This creates a recovery path without a centralized appeals process. The system is self-governing.</p><h2 id="frequently-asked-questions">Frequently Asked Questions</h2><p><strong>What is a Sparse Merkle Tree in simple terms?</strong></p><p>A Sparse Merkle Tree is a fixed-size cryptographic tree where every possible leaf position exists from the start. Empty positions store a default zero hash. This design allows efficient proofs of both presence and absence without scanning the entire tree.</p><p><strong>Why does RLN on Status Network use a height-20 Sparse Merkle Tree?</strong></p><p>Height 20 creates exactly 1,048,576 leaf slots, enough to register 1 million accounts. This fixed size also ensures every ZK proof path is exactly 20 hashes long, which is required for the ZK circuit to have a consistent, compile-time structure.</p><p><strong>What is a non-membership proof and why does it matter for rate limiting?</strong></p><p>A non-membership proof cryptographically demonstrates that a value does not exist in the tree. In rate limiting, this allows the system to verify that a user has not already consumed their quota for a given epoch, without revealing any details about other users or their activity.</p><p><strong>How does Shamir's Secret Sharing work with RLN to punish spammers?</strong></p><p>Each message a user sends reveals one share of their private key, derived using Shamir's Secret Sharing. Staying within quota reveals too few shares to reconstruct the key. Exceeding quota reveals enough shares to reconstruct and expose the key, triggering removal from the system.</p><p><strong>Can a user recover from being added to the RLN Deny List on Status Network?</strong></p><p>Yes. Users on the Deny List can earn Karma back through contributions such as SNT staking, bridging assets, or providing liquidity. Accumulating sufficient Karma can restore their gasless transaction quota over time, without requiring any centralized intervention.</p><p><strong>Why are Sparse Merkle Trees better than standard Merkle Trees for ZK proofs?</strong></p><p>Standard Merkle Trees have variable depth depending on how many leaves are populated. ZK circuits require a fixed structure at compile time.</p><p>Sparse Merkle Trees have a constant height regardless of how many leaves are filled, making every proof path the same length and compatible with ZK proving systems.</p><p><strong>What is a nullifier in the context of RLN rate limiting?</strong></p><p>A nullifier is a deterministic output produced when a user sends a message within a given epoch. If the same user sends two messages in the same epoch, both messages produce the same nullifier. Duplicate nullifiers reveal the secret key shares needed to identify and ban the account.</p><p><strong>Does the RLN Sparse Merkle Tree store user identities on-chain?</strong></p><p>No. The SMT stores cryptographic commitments derived from users' secret keys, not identities.
